= Lab Guide: Install and Verify Signed Collections

_A guide to installing signed collections with GPG signature verification and detecting content tampering._

---

[discrete]
====
*Estimated time to complete: 15 minutes*
====

In the previous module, you signed and approved collections in Private Automation Hub. In this final module, you will install those signed collections and learn how to verify their signatures on the client side using `ansible-galaxy`.

This module uses a *different GPG key* than the one you created in Module 2. In Module 2, you created a personal key to sign projects locally with `ansible-sign`. Here, you will use the *hub's signing service public key* to verify collections that were signed server-side by PAH.

---

== Task 1: Import the Hub's Public Key into a Keyring

The hub's signing service public key was exported to your home directory during setup. Import it into a dedicated GPG keyring:

[source,bash]
----
gpg --import --no-default-keyring --keyring ~/keyring.kbx ~/galaxy_signing_service.asc
----

This creates a keyring file at `~/keyring.kbx` containing the hub's public key. The `ansible-galaxy` CLI will use this keyring to verify collection signatures.

Now set the trust level on the imported key so GPG considers it fully trusted for verification:

[source,bash]
----
KEY_FP=$(gpg --no-default-keyring --keyring ~/keyring.kbx --with-colons --list-keys | awk -F: '/^fpr/{print $10; exit}')
echo "${KEY_FP}:6:" | gpg --no-default-keyring --keyring ~/keyring.kbx --import-ownertrust
----

You can verify the key was imported and trusted:

[source,bash]
----
gpg --no-default-keyring --keyring ~/keyring.kbx --list-keys
----

---

== Task 2: Install a Collection Without Signature Verification

First, install `ansible.test_collection` *without* specifying a keyring to see what happens:

[source,bash]
----
cd ~
ansible-galaxy collection install ansible.test_collection -c
----

The collection installs successfully with no errors. However, without the `--keyring` option, `ansible-galaxy` skips signature verification entirely -- it has no way to confirm that the collection is authentic and untampered. The install succeeds, but you have no cryptographic guarantee about the content you just downloaded.

---

== Task 3: Install a Collection With Signature Verification

Now install `community.lab_collection` with signature verification enabled:

[source,bash]
----
ansible-galaxy collection install community.lab_collection --keyring ~/keyring.kbx -c -vvvv
----

In the verbose output, look for:

[source,text]
----
GnuPG signature verification succeeded for community.lab_collection
----

This confirms that `ansible-galaxy` downloaded the collection, fetched its signature from the hub, and verified it against the public key in your keyring. The collection's integrity is cryptographically guaranteed.

---

== Task 4: Examine the Signature Metadata

After installation, `ansible-galaxy` stores signature metadata alongside the collection. Inspect it:

[source,bash]
----
cat ~/.ansible/collections/ansible_collections/community.lab_collection-1.0.0.info/GALAXY.yml
----

This file contains the collection version metadata including the signatures that were applied by the hub's signing service.

---

== Task 5: Tamper With the Installed Collection

To demonstrate integrity verification, add an unexpected file to the installed collection:

[source,bash]
----
touch ~/.ansible/collections/ansible_collections/community/lab_collection/docs/README.md
----

This simulates a scenario where someone (or something) has modified a collection after it was installed -- for example, a compromised dependency or an accidental edit.

---

== Task 6: Verify Collection Integrity

Now run `ansible-galaxy collection verify` to compare the local installation against the remote manifest:

[source,bash]
----
cd ~
ansible-galaxy collection verify community.lab_collection --keyring ~/keyring.kbx -c --server published_repo
----

The output will report the tampered file:

[source,text]
----
Collection community.lab_collection contains modified content in the following files:
    docs/README.md
----

The verification process:

1. Checks the GPG signature on the remote manifest to confirm it was signed by a trusted key.
2. Compares the local files against the remote checksums.
3. Reports any differences -- added, modified, or removed files.

---

== Summary

* You imported the hub's signing service public key into a local GPG keyring and set it as trusted.
* You installed a collection *without* verification and saw that it succeeds silently -- with no integrity guarantee.
* You installed a collection *with* verification and confirmed the signature was valid.
* You tampered with an installed collection and used `ansible-galaxy collection verify` to detect the modification.
